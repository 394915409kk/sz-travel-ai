# 深圳职旅 Codex 总规则

## 1. 项目定位

本项目是“深圳市职工国际旅行社 AI 数字员工系统”的 Codex 能力中心，负责沉淀 AGENTS.md、skills、agents、prompts、checklists、playbooks、scripts、references 等工作资产。

本项目只建设 Codex 执行能力、工作流标准、提示词模板、风控规则、验收清单和 skill 目录，不开发业务主系统功能，不连接真实密码、付款权限、生产数据库或客户隐私数据。

## 2. Codex 工作总原则

- 先读上下文，再动手：涉及代码、规则或业务判断时，先检查现有文件、约束和历史规则。
- 小步闭环：每次只改与任务相关的文件，完成后给出可验收结果。
- 证据优先：业务结论、价格、库存、航班、签证政策、利润率、现金流、ROI、客户数据必须有来源；没有来源时标记为 `待补充`、`待核价`、`待实测`、`待人工确认`。
- 保持边界：不把能力中心当作业务生产系统，不在本项目保存真实客户隐私、供应商底价、支付凭证、密钥或生产库连接。
- 尊重现有改动：不要回滚用户或其他协作者的未提交内容，除非用户明确要求。
- 输出可交付：默认给老板、工程师、销售或运营可直接使用的版本，而不是只给概念说明。

## 3. 深圳职旅业务板块

本能力中心默认服务以下业务板块：

- CEO 经营总控：利润、现金流、转化率、履约风险、客户满意度、员工绩效。
- 产品中心：塞班、富国岛、入境中国、国内定制、企业团建、疗休养、MICE、亲子与中老年产品。
- 销售与私域：客服接待、线索分级、报价跟进、复购转介绍、B2B 同业、企业客户。
- 供应链：机票、酒店、地接、车导、景区、保险、签证、OTA 与渠道资源。
- 定价与利润：成本拆分、动态报价、毛利测算、库存风险、收款节点。
- 财务风控：应收应付、对账、现金流预警、退款、坏账、预算控制。
- 合同与合规：供应商合同、客户合同、OTA 规则、投诉处理、用印审批。
- 内容增长：小红书、抖音、视频号、朋友圈、Facebook、Instagram、TikTok、B2B 宣发。
- 技术运维：FastAPI、SQLite、前端页面、自动化脚本、GitHub PR、测试和部署前检查。

## 4. 任务完成标准

Codex 完成任务前必须确认：

- 目标已落到明确文件、模板、清单、脚本或代码改动中。
- 业务类输出没有编造价格、库存、政策、利润、客户数据或供应商承诺。
- 技术类输出通过可运行检查，或明确说明未能运行的原因。
- 需要人工补充的信息已列成清单，责任人和用途清楚。
- 对老板侧输出，优先呈现结论、表格、风险和下一步行动。
- 对工程侧输出，说明改动范围、验证方式、剩余风险和后续建议。

## 5. 风控红线

Codex 不得执行或暗示执行以下行为：

- 保存、索取或使用真实密码、密钥、银行卡、付款验证码、生产数据库凭据。
- 连接生产数据库、真实支付系统、真实客户隐私库，除非用户另行建立合规授权环境。
- 编造航班、酒店库存、签证政策、报价、成本、利润、ROI、合同条款或平台规则。
- 对法律、税务、财务审计、保险理赔作最终专业意见；只能输出风险提示和待法务/财务确认项。
- 自动发送客户消息、供应商承诺、付款指令、退款承诺或合同定稿。
- 自动 merge、push、发布生产环境，除非用户明确授权当前步骤。

## 6. 数据安全规则

- 使用脱敏样例数据进行模板、脚本和测试。
- 客户姓名、手机号、证件号、护照号、生日、住址、微信号、支付信息必须脱敏。
- 供应商底价、账期、返佣、返点、合同敏感条款默认不进入公开文档。
- 任何可复用模板都应使用占位符，如 `客户姓名_待补充`、`成本_待核价`。
- 输出文件不得包含真实生产连接串、密钥、cookie、token 或授权头。

## 7. GitHub / PR / 测试要求

- 开始工程任务前确认仓库根目录、当前分支、远端和工作区状态。
- 用户要求“不要 commit / 不要 push / 不要 merge”时必须严格遵守。
- 代码改动应附带与风险匹配的测试、验收清单或人工检查步骤。
- PR 描述应包含：背景、改动范围、验证结果、风险与回滚说明。
- 数据库迁移必须包含备份、回滚或兼容说明，禁止无确认删除生产数据。
- 修复 bug 时优先复现，再修复，再验证；无法复现时说明已检查的证据。
